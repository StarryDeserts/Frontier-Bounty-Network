# Why Smart Gate First

## Why Smart Gate is the right first slice

Smart Gate is the best first demo slice because it is the shortest path from bounty state to visible infrastructure behavior.

It answers the judge question immediately:
- why does this belong in EVE Frontier?

The answer is:
- because wanted-player state becomes programmable infrastructure policy.

## Why not start with the full kill-record provider

A full kill-record provider path is important, but it is not the best first hackathon demo.

Why not first:
- it is harder to explain in 3 minutes
- it depends on external provider integration and verifier trust design
- much of the complexity is back-end and trust plumbing rather than visible gameplay meaning
- judges see less immediate world impact from it

By contrast, Smart Gate gives a clean story:
- a player is wanted
- a gate reacts
- infrastructure behavior changes

## Why Smart Gate is better for a hackathon demo

It is better for a demo because:
- the concept is legible in one sentence
- the behavior is easy to visualize
- it maps directly to EVE Frontier Smart Infrastructure language
- it feels like a real gameplay and service-economy mechanic, not just a backend architecture slide

It also lets the demo stay honest:
- the policy surface is real and on-chain
- the final runtime hookup is staged but understandable

## How it expands naturally

Smart Gate first does not limit the product. It creates the cleanest expansion path.

Next natural extensions:
- Smart Turret consumes the same wanted signal for threat prioritization
- kill-record provider completes the settlement side of the loop
- verifier hardening upgrades trust in the claim path

So Smart Gate first is not a detour.
It is the clearest entry point into the full bounty-powered infrastructure story.
